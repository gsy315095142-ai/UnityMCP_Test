## 📋 项目概述

Unity-MCP 是一个成熟的、已经在生产环境中使用的 Unity 编辑器扩展项目,它通过 **MCP (Model Context Protocol)** 协议将 Unity 编辑器与 AI 客户端(Claude、Cursor、Windsurf 等)连接起来,实现 AI 辅助游戏开发。

### 核心特点

1. **完整的 MCP 实现** - 遵循标准 MCP 协议规范
2. **三层架构** - Unity Plugin、MCP Server、MCP Client 清晰分离
3. **Runtime 和 Editor 双模式** - 不仅支持编辑器,还支持运行时游戏内 AI
4. **丰富的内置工具** - 60+ 开箱即用的 MCP Tools
5. **高度可扩展** - 支持自定义 MCP Tool、Resource、Prompt

### 三层架构

```
┌─────────────────────────────────────────────────────────────┐
│                     MCP Client (AI 客户端)                   │
│   Claude Code / Claude Desktop / Cursor / Windsurf / ...    │
│                  (用户与AI对话的界面)                         │
└───────────────────────┬─────────────────────────────────────┘
                        │ MCP Protocol (stdio / streamableHttp)
                        ↓
┌─────────────────────────────────────────────────────────────┐
│                    Unity-MCP-Server                          │
│              (独立的 .NET 服务器进程)                         │
│   - 处理 MCP 协议通信                                         │
│   - 转发请求到 Unity Plugin                                   │
│   - 支持 Docker 部署和本地运行                                │
└───────────────────────┬─────────────────────────────────────┘
                        │ SignalR / HTTP (端口 8080)
                        ↓
┌─────────────────────────────────────────────────────────────┐
│                  Unity-MCP-Plugin                            │
│               (Unity 编辑器插件)                              │
│   - 注册和暴露 MCP Tools                                      │
│   - 执行 Unity API 调用                                       │
│   - 管理项目资源和场景                                         │
└─────────────────────────────────────────────────────────────┘
```

### 为什么需要独立的 MCP Server?

1. **协议兼容性** - Unity 进程无法直接与 MCP Client 通过 stdio 通信
2. **隔离性** - Server 崩溃不会影响 Unity 编辑器
3. **灵活部署** - 支持本地和云端部署
4. **多 Client 支持** - 一个 Server 可以同时服务多个 MCP Client

## 💡 核心设计模式

### 1. Tool 定义模式

通过 **Attribute 驱动** 的方式定义 MCP Tools:

```csharp
[McpPluginToolType]  // 标记类包含 MCP Tools
public partial class Tool_GameObject
{
    [McpPluginTool("gameobject-create", Title = "GameObject / Create")]
    [Description("Create a new GameObject...")]
    public GameObjectRef Create(
        [Description("Name of the new GameObject.")]
        string name,

        [Description("Parent GameObject reference...")]
        GameObjectRef? parentGameObjectRef = null,  // 可选参数用 ? 和默认值

        Vector3? position = null,
        Vector3? rotation = null,
        Vector3? scale = null
    )
    {
        // 关键:在主线程执行 Unity API
        return MainThread.Instance.Run(() =>
        {
            var go = new GameObject(name);
            // ... Unity API 调用
            return new GameObjectRef(go);
        });
    }
}
```

**关键点**:
- ✅ 使用 `Description` 特性为 AI 提供参数说明
- ✅ 可选参数用 `?` + 默认值标记
- ✅ 所有 Unity API 必须在主线程执行
- ✅ 返回类型支持序列化(GameObjectRef、Vector3 等)

### 2. Roslyn 动态代码执行

最强大的功能之一 - 即时编译和执行 C# 代码:

```csharp
[McpPluginTool("script-execute", Title = "Script / Execute")]
[Description("Compiles and executes C# code dynamically using Roslyn.")]
public static SerializedMember? Execute(
    [Description("C# code that compiles and executes immediately...")]
    string csharpCode,

    string className = "Script",
    string methodName = "Main",
    SerializedMemberList? parameters = null
)
{
    // 1. 使用 Roslyn 编译代码
    var compilation = CSharpCompilation.Create(
        assemblyName: "DynamicAssembly",
        syntaxTrees: new[] { CSharpSyntaxTree.ParseText(code) },
        references: AssemblyUtils.AllAssemblies
            .Where(a => !a.IsDynamic)
            .Select(a => MetadataReference.CreateFromFile(a.Location))
            .ToArray()
    );

    // 2. 编译到内存
    using (var ms = new MemoryStream())
    {
        var result = compilation.Emit(ms);
        if (!result.Success)
            throw new Exception($"Compilation failed: {errors}");

        // 3. 加载并执行
        ms.Seek(0, SeekOrigin.Begin);
        var assembly = Assembly.Load(ms.ToArray());
        var type = assembly.GetType(className);
        var method = type.GetMethod(methodName);
        return method.Invoke(null, parsedParameters);
    }
}
```

**作用**:
- AI 可以生成临时代码并立即执行,无需等待 Unity 编译
- 实现快速迭代和测试
- 适合一次性操作和脚本化任务

### 3. Reflection 工具

通过反射暴露整个 Unity API:

```csharp
[McpPluginTool("reflection-method-find")]
public static MethodInfo FindMethod(string typeName, string methodName);

[McpPluginTool("reflection-method-call")]
public static object CallMethod(string typeName, string methodName, params object[] args);
```

**意义**:
- AI 可以调用任何 Unity API,即使没有专门的 Tool
- 支持调用私有方法(测试和调试用)
- 可以访问项目代码中的自定义方法

### 4. 序列化系统

统一的对象引用和序列化机制:

```csharp
// GameObjectRef - 引用 GameObject
public class GameObjectRef
{
    public string? scenePath;      // 场景路径
    public string? hierarchyPath;  // 层级路径
    public int? instanceId;        // 实例ID

    public GameObject FindGameObject(out string? error)
    {
        // 通过多种方式定位对象
    }
}

// SerializedMember - 通用序列化
public class SerializedMember
{
    public string TypeName;
    public object Value;
    public Dictionary<string, SerializedMember> Fields;
}
```

**优势**:
- AI 可以引用和操作场景中的现有对象
- 支持复杂对象的传递和修改
- 保持类型信息,避免序列化丢失

## 🔧 关键技术点

### 1. 主线程调度

Unity API 只能在主线程调用,项目使用自定义的 `MainThread` 工具:

```csharp
public class MainThread : MonoBehaviour
{
    public static MainThread Instance { get; private set; }

    public T Run<T>(Func<T> action)
    {
        // 将操作调度到主线程执行
        // 使用 TaskCompletionSource 同步等待结果
    }
}
```

### 2. 自动配置系统

支持多种 MCP Client 的自动配置:

```csharp
public abstract class AiAgentConfiguratorBase
{
    public abstract string Name { get; }
    public abstract void Configure(string jsonConfig);
}

// 具体实现
public class ClaudeCodeConfigurator : AiAgentConfiguratorBase
{
    public override void Configure(string jsonConfig)
    {
        // 通过 CLI 命令自动配置
        // claude mcp add ai-game-developer <command>
    }
}
```

### 3. 热更新 Tools

MCP Server 支持动态更新 Tools 列表:

- Unity Plugin 启动时扫描所有带 `[McpPluginToolType]` 的类
- 通过 SignalR 推送到 MCP Server
- MCP Server 通知所有连接的 MCP Client
- AI 立即获得新的 Tool 定义

### 4. 错误处理

统一的错误反馈机制:

```csharp
try
{
    // Unity API 调用
    var go = new GameObject(name);
}
catch (Exception ex)
{
    throw new ArgumentException(
        $"Failed to create GameObject '{name}': {ex.Message}",
        nameof(name)
    );
}
```

返回详细的错误信息给 AI,帮助其自我纠正。

## 🎯 可借鉴的设计亮点

### 1. **分层架构** ⭐⭐⭐⭐⭐

**优点**:
- 职责清晰,易于维护
- Server 独立部署,支持云端和本地
- Plugin 和 Server 可以独立更新

**我们的借鉴**:
- 如果只做本地开发,可以简化为两层(Plugin + 内嵌 Server)
- 如果需要团队协作,建议保持三层架构

### 2. **Attribute 驱动的 Tool 定义** ⭐⭐⭐⭐⭐

**优点**:
- 开发者友好,无需手写 JSON Schema
- 类型安全,编译时检查
- 自动生成文档和 Tool 描述

**我们的借鉴**:
- **强烈推荐采用这种模式**
- 可以基于 Reflection 自动发现和注册 Tools

### 3. **Roslyn 动态执行** ⭐⭐⭐⭐

**优点**:
- 极大提升 AI 灵活性
- 无需预定义所有可能的操作
- 支持快速原型和一次性任务

**我们的借鉴**:
- 初期可以不实现,专注核心功能
- 中后期添加,作为"兜底"能力

### 4. **Reflection API** ⭐⭐⭐⭐

**优点**:
- 暴露整个 Unity API 和项目代码
- 无需为每个 Unity API 写 Tool
- 支持访问私有成员(调试用)

**我们的借鉴**:
- 建议实现基础版本(公共方法调用)
- 安全性考虑:可以限制可调用的命名空间

### 5. **统一序列化系统** ⭐⭐⭐⭐⭐

**优点**:
- 支持对象引用传递
- 保留类型信息
- 处理 Unity 特殊类型(GameObject、Component 等)

**我们的借鉴**:
- **必须实现**
- 可以基于 Unity 的 JsonUtility + 自定义 Converter

### 6. **多 MCP Client 支持** ⭐⭐⭐

**优点**:
- 用户可以选择喜欢的 AI 工具
- 不绑定特定平台

**我们的借鉴**:
- 初期可以只支持 Claude Code
- 预留扩展接口

## ⚠️ 潜在问题和局限

### 1. 复杂度较高

三层架构增加了部署和调试复杂度:
- 需要单独启动 MCP Server
- 网络通信可能出现连接问题
- 端口占用和防火墙问题

### 2. 性能考虑

每个 Tool 调用都涉及多次跨进程通信:
```
MCP Client → MCP Server → Unity Plugin → Unity API
```

**影响**: 延迟较高(通常 50-200ms)

### 3. 序列化限制

某些 Unity 对象无法序列化:
- 事件(Event)
- 委托(Delegate)
- 非公共字段

## 🎯 总结

Unity-MCP 是一个**架构优秀、功能完整、生产可用**的参考项目。

### 最值得学习的三点:

1. **Attribute 驱动的 Tool 定义** - 开发体验极佳
2. **统一的序列化和引用系统** - 解决了核心技术难题
3. **分层架构** - 清晰的职责划分

### 我们的定位差异:

| 特性 | Unity-MCP | 我们的项目 |
|------|-----------|-----------|
| **目标** | 通用 Unity 开发助手 | 专注代码+预制体生成 |
| **复杂度** | 高(60+ Tools,三层架构) | 中(20-30 Tools,简化架构) |
| **适用场景** | 全流程开发 | 快速原型和资源生成 |
| **Runtime 支持** | 是 | 否(初期) |
| **部署方式** | 本地 + 云端 + Docker | 仅本地(初期) |

### 推荐路线:

1. **借鉴架构,简化实现** - 学习设计模式,但从小做起
2. **专注核心价值** - 代码生成 + 预制体创建,做精做深
3. **快速迭代** - 先做 MVP,收集反馈,再扩展
4. **保持扩展性** - 预留接口,方便后续添加功能
